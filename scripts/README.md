# Scripts Layout

根目录只保留高频手动入口：

- `code_compile_check.py`
- `code_runtime_check.py`
- `code_format.py`
- `release_check.py`
- `setup_env.py`
- `start_app.py`

其中 `release_check.py --scope multi-display` 适合本地快速串起多屏 compile/runtime/doc 回归；runtime scope 现在会按 `EGUI_CONFIG_MAX_DISPLAY_COUNT` 逐屏校验录制帧是否成套输出，校验多屏示例的关键录制阶段标签是否完整，并覆盖 `HelloMultiDisplay` 的主屏点击隔离与双屏并发 activity 动画、`HelloMultiDisplayHetero` 的副屏 tick 连续性/点击复位，以及退出摘要里的线程回收、SDL 窗口销毁顺序和 core task queue 指标。如果一时记不清某个 scope 会跑什么，直接执行 `python scripts/release_check.py --help`，底部的 `Scoped profiles` 会列出 scope 摘要；`multi-display` 当前支持的 `--only` 值是 `format, compile, runtime, doc`。实际运行 `python scripts/release_check.py --scope multi-display` 时，开头会直接打印这些 step filters、当前激活的 `--only/--skip` 过滤结果、可拆看的 compile/runtime/doc drill-down 命令和关键产物目录；失败摘要末尾会重复列出“只重跑当前失败步骤”的 `python scripts/release_check.py --scope multi-display --only <step>`、对应底层命令与产物位置，通过时也会把已完成步骤的关键产物目录再汇总一遍。若要做低容量队列探针，可直接用 `python scripts/release_check.py --scope multi-display --only compile,runtime --queue-capacity-probe 1`；若要做 burst core-task 压力探针，可用 `python scripts/release_check.py --scope multi-display --only compile,runtime --queue-stress-probe 8`；若要探测 backpressure 配置本身，可加 `--queue-post-retry-probe N` / `--queue-post-retry-delay-probe N`；若要把 backpressure 也变成显式约束，可加 `--max-core-task-retries N`、`--max-single-post-core-task-retries N` 或 `--fail-on-full-core-task-queue`。这些参数可以组合使用，用来验证“温和 workload 能过，但 burst 场景下会不会靠高 retries、单次尖峰重试或队列打满硬扛”的边界。

低频或专项脚本按用途分目录：

- `checks/`
  专项校验与回归脚本，如 icon font、dirty animation、parity、render workflow、touch 语义检查。
  `multi_display_core_task_stress_scan.py` 可批量扫描 multi-display 的 `queue capacity / burst stress` 组合边界，并输出通过阈值摘要；也可直接用 `--retry-counts` 扫描 `retry_count` 这一维、用 `--retry-delay-values` 扫描 `retry_delay_ms` 这一维、用 `--stress-post-gap-values` 扫描 synthetic burst 相邻 post 之间的 `gap_ms` 这一维、用 `--max-core-task-retry-values` 扫描累计 retries guardrail 这一维、用 `--max-single-post-core-task-retry-values` 扫描单次 post 尖峰重试阈值这一维、用 `--fail-on-full-core-task-queue-values` 扫描 full-queue guard 这一维，或透传 `--queue-post-retry-probe` / `--queue-post-retry-delay-probe` / `--queue-stress-post-gap-probe`、`--max-core-task-retries` / `--max-single-post-core-task-retries` / `--fail-on-full-core-task-queue`，把 backpressure 配置和成本一起纳入扫描结果。若不想每次手敲整组二维参数，可直接用 `--profile retry-delay`、`--profile delay-gap`、`--profile retry-gap` 这三个高频模板；profile 只会填充未显式传入的扫描维度，不会覆盖你手动指定的值。若只想看某一个示例，可直接加 `--app HelloMultiDisplay` 或 `--app HelloMultiDisplayHetero`，scan 会把 compile/runtime 都收敛到单 app，并额外输出 `Per-App Threshold Summary`，直接给出该 app 在每个 `capacity/burst` 下的最小通过 `retry/delay/gap/max-core/max-single-post/full-queue` 组合，以及 `dominant_stage`、`stage_failures`、`progress_range`、`dominant_contention`、`contention_failures`；如果当前扫描维度本身能生成 grid，还会继续输出 `Per-App Delay/Gap Grid Summary`、`Per-App Retry/Delay Grid Summary`、`Per-App Retry/Gap Grid Summary`，把单 app 失败格点进一步压成 `FAIL:<contention>@<stage>`，去掉重复的 app 前缀，便于直接对比同一个 app 的边界形状。若想把单 app 输出再压紧，可叠加 `--compact-app-view`，这时脚本会保留 per-app 摘要和通用 stage/contention 统计，但不再重复输出通用 app-prefixed grid 与 `Failure App *` 段落。扫描末尾还会输出 `Suggested Repro Commands`，直接给出当前扫描的重跑命令，以及单 app 模式下的 `replay_minimum_passing` / `replay_first_failure` 命令，并补一行 `replay_first_failure_context` 标出首个失败的 `app/stage/progress/contention`；若开启 `--exit-on-first-failure`，脚本会在首个 compile/runtime 失败后立即停止继续枚举后续组合，但仍保留当前已收集结果的文本摘要、JSON summary、repro JSON 与 JSONL 导出，并在机器输出里标记 `stopped_early` / `stop_reason`，同时补充 `planned_combinations` / `executed_combinations` / `remaining_combinations` 方便 CI 判断是否提前收口及剩余覆盖范围；若再叠加 `--emit-json-summary`，还会输出一份机器可读 JSON，包含扫描维度、结果列表、per-app threshold、repro commands 和 `replay_first_failure_context`；若改用 `--emit-json-summary-compact`，则会在文本报告末尾追加一行紧凑 JSON；若改用 `--json-summary-file <path>`，则会把同一份 JSON 直接写到指定文件，相对路径按仓库根目录解析；若改用 `--emit-suggested-repro-json`，则会把建议重跑命令和 `replay_first_failure_context` 独立输出成 JSON；若改用 `--emit-suggested-repro-json-compact`，则会把同一份 repro JSON 直接压成单行输出；若改用 `--suggested-repro-json-file <path>`，则会把同一份 repro JSON 直接写到指定文件；若改用 `--results-jsonl-file <path>`，则会把归一化后的逐条扫描结果写成 JSONL，便于 `jq`、日志采集或 CI 后处理，并可继续用 `--results-jsonl-status all|pass|fail|compile-fail|runtime-fail` 只导出所需状态；若还想继续缩窄结果集，可再叠加 `--results-jsonl-capacity <n>`、`--results-jsonl-burst <n>`、`--results-jsonl-stage <stage>`、`--results-jsonl-contention <contention>`、`--results-jsonl-progress <progress>` 做精确匹配过滤，或改用 `--results-jsonl-progress-range <start/end>` 按数值进度区间过滤，并继续按 `--results-jsonl-retry/--results-jsonl-delay/--results-jsonl-gap` 与 `--results-jsonl-max-core-retries/--results-jsonl-max-single-post-retry/--results-jsonl-full-queue-guard` 精确过滤扫描维度；现在每一条 result JSONL 也会同步携带 `failure_kind`、`progress_step`、`progress_total`、`combination_key`、`combination_rank`、`has_stage_tag`、`has_contention_tag`，便于直接分组聚合、数值排序或快速判断是否带标签，并继续携带 `stopped_early` / `stop_reason` / `planned_combinations` / `executed_combinations` / `remaining_combinations`；若改用 `--emit-results-jsonl`，则会直接把同一批 JSONL 结果打印到 stdout，适合管道消费；若改用 `--per-app-thresholds-jsonl-file <path>`，则会把 `Per-App Threshold Summary` 同步导出为 JSONL，便于阈值表或趋势看板直接消费，并可再用 `--per-app-thresholds-status all|has-pass|no-pass` 过滤；若还想继续缩窄阈值行，可再叠加 `--per-app-thresholds-capacity <n>`、`--per-app-thresholds-burst <n>`、`--per-app-thresholds-stage <stage>`、`--per-app-thresholds-contention <contention>`、`--per-app-thresholds-progress-range <range>` 按 `dominant_stage/dominant_contention/progress_range` 做精确匹配，或改用 `--per-app-thresholds-progress-overlap <start/end>` 按进度窗口相交关系过滤，并继续按 `--per-app-thresholds-min-retry/--per-app-thresholds-min-delay/--per-app-thresholds-min-gap` 与 `--per-app-thresholds-min-max-core-retries/--per-app-thresholds-min-max-single-post-retry/--per-app-thresholds-min-full-queue-guard` 精确过滤最小通过组合；现在每一条 per-app threshold JSONL 也会同步携带 `scope_key`、`scope_rank`、`minimum_passing_key`、`minimum_passing_rank`、`has_passing`、`minimum_passing_retry`、`minimum_passing_delay`、`minimum_passing_gap`、`minimum_passing_max_core_task_retries`、`minimum_passing_max_single_post_retry`、`minimum_passing_full_queue_guard`、`result_count`、`pass_count`、`fail_count`、`all_passed`、`all_failed`、`progress_min_step`、`progress_max_step`、`progress_total` 与同一组提前停止和覆盖率元字段；若改用 `--emit-per-app-thresholds-jsonl`，则会把同一批 threshold JSONL 直接打印到 stdout。`--queue-stress-post-gap-probe` 用于固定设置一个 gap，`--stress-post-gap-values` 则用于批量找出最小可通过 gap，便于把“burst 自己的 self-contention”与“业务阶段本身的外部 contention”拆开看。脚本会先对各扫描值去重并按升序归一化，避免传参乱序或重复值时把 `minimum/maximum` 阈值摘要算偏；若运行时失败信息里已经带了 recording stage 标签，还会额外输出 `Failure App Summary`、`Failure App Rollup`、`Failure App Leaders`、`Failure Stage Summary`、`Failure Stage Rollup`、`Failure Stage Leaders`、`Failure Contention Summary`、`Failure Contention Rollup` 和 `Failure Contention Leaders`，分别用于逐条列出 `app=...`、按 app 汇总 `failures=N`、指出每个扫描组合当前的 `dominant_app=...`，以及逐条列出 `stage=... progress=i/N`、按阶段汇总 `failures=N progress_range=...`、指出每个扫描组合当前的 `dominant_stage=...`，并进一步区分 `self_inflight / foreign_inflight / self_pending / foreign_pending` 这类“谁在占槽”的失败模式。若固定 `delay/gap/max-core/max-single-post/full-queue` 只扫描 `--retry-counts`，摘要会直接给出 `minimum passing retry=...`；若固定 `retry/gap/max-core/max-single-post/full-queue` 只扫描 `--retry-delay-values`，摘要会直接给出 `minimum passing delay=...`；若固定 `retry/delay/max-core/max-single-post/full-queue` 只扫描 `--stress-post-gap-values`，摘要会直接给出 `minimum passing gap=...`；若固定 `retry/max-core/max-single-post/full-queue` 同时扫描 `--retry-delay-values` 与 `--stress-post-gap-values`，摘要会继续补出“每个 delay 对应的 minimum passing gap”和“每个 gap 对应的 minimum passing delay”，并在 `capacity/burst` 固定时额外输出 `Delay/Gap Grid Summary`，把每个格点压成 `PASS` 或 `FAIL:<app>:<contention>@<stage>`；若固定 `gap/max-core/max-single-post/full-queue` 同时扫描 `--retry-counts` 与 `--retry-delay-values`，会额外输出 `Retry/Delay Grid Summary`；若固定 `delay/max-core/max-single-post/full-queue` 同时扫描 `--retry-counts` 与 `--stress-post-gap-values`，会额外输出 `Retry/Gap Grid Summary`；若固定 `retry/delay/gap/max-single-post/full-queue` 只扫描 `--max-core-task-retry-values`，摘要会直接给出 `minimum passing max_core_task_retries=...`；若固定 `retry/delay/gap` 只扫描 `--max-single-post-core-task-retry-values`，摘要会直接给出 `minimum passing max_single_post_retry=...`；若固定 `retry/delay/gap/max-core/max-single-post` 只扫描 `--fail-on-full-core-task-queue-values`，摘要会直接给出 `minimum passing full_queue_guard=...`。脚本默认会对非确定性 compile/runtime 失败做一次有限重试，减少偶发噪声。
  新增 rank 字段后，也支持用 `--results-jsonl-combination-rank`、`--per-app-thresholds-scope-rank`、`--per-app-thresholds-min-passing-rank` 直接过滤。
  现也支持按布尔标签直接过滤：`--results-jsonl-has-stage-tag`、`--results-jsonl-has-contention-tag`、`--per-app-thresholds-all-passed`、`--per-app-thresholds-all-failed`。
- `platform/`
  平台/工程维护脚本，如 `keil_project_sync.py`、`update_visual_studio_sln.py` 和 `visual_studio_project_check.py`。新增、删除或重命名 Visual Studio 工程内源码后，先运行 `python scripts/platform/update_visual_studio_sln.py` 同步；发布前可用 `python scripts/release_check.py --only visual_studio` 检查 `.sln/.vcxproj/.filters` 是否最新并做轻量 MSBuild 编译。
- `web/`
  Emscripten / WASM 相关脚本，如 `wasm_build_demos.py`、`emcc_wrapper.py`。
- `recording/`
  录制与 GIF 导出辅助脚本。

整理原则：

- 高频、面向大多数开发者的入口留在根目录。
- 专项验证、平台专用、发布打包、历史兼容脚本进入子目录。
- 如果某类能力后续完全迁出仓库，对应子目录优先整体删除，而不是重新回到根目录。
