# RadialMenu 控件设计说明

## 1. 为什么需要这个控件

radial_menu 面向“小屏幕 + 快速上下文操作”场景，目标是在有限空间中提供比传统线性菜单更高的信息密度和更直接的方向感。它适合以下典型情况：

- 圆形表盘、仪表盘、地图、画布等中心区域已经存在主内容，不适合再挤出一整条菜单栏。
- 需要单手操作，用户希望通过“按下中心 -> 朝目标方向滑出 -> 松手确认”的动作快速完成选择。
- 需要明显新颖的控件形态，为后续 1000 控件计划建立更强的视觉辨识度样板。

该控件优先强调“极坐标布局、扇区高亮、拖拽释放确认”三点，而不是做成普通菜单的圆形皮肤。

## 2. 为什么现有控件还不够

现有控件不能直接替代 radial_menu：

- menu：偏线性列表或层级展开，主要优势是顺序浏览，不具备方向式选择体验。
- button_matrix：适合矩形规则排布，但不提供中心触发区、极坐标命中或扇区滑选语义。
- tab_bar：定位是页面导航，不适合作为临时上下文动作面板。
- compass / arc_slider：虽然有圆弧或角度表达，但目标是显示与调节，不是多动作菜单。

因此 radial_menu 的非替代性边界应明确为：

1. 以中心触发为入口。
2. 以扇区作为动作单元。
3. 以拖拽方向选择和松手确认作为核心交互。

## 3. 应用概览与使用场景

本次 HelloCustomWidgets 示例计划展示 3 组 radial_menu：

- radial_primary：主示例，6 个动作项，完整展示展开、悬停、确认、取消。
- radial_compact：紧凑示例，4 个动作项，验证小尺寸下的布局与文本裁切策略。
- radial_disabled：禁用示例，验证控件在只读或不可交互状态下的视觉降级与点击无效。

目标使用场景：

- 表盘页面上的快捷操作入口，如“记录 / 暂停 / 标记 / 导出”。
- 地图或画布页面上的上下文动作，如“定位 / 缩放 / 标注 / 图层”。
- 车载、穿戴或设备调试界面中的单手快速选择。

控件目录：

- example/HelloCustomWidgets/navigation/radial_menu/
## 4. 视觉与布局规格

屏幕基准：240 x 320。

整体布局建议：

- 根容器：linearlayout，纵向排列，整体水平居中。
- 标题标签区域：220 x 24，用于说明当前交互目标。
- radial_primary：184 x 184，6 个扇区，中心按钮直径约 44，外圈半径约 82。
- radial_compact：148 x 148，4 个扇区，中心按钮直径约 36，外圈半径约 66。
- radial_disabled：184 x 184，6 个扇区，整体透明度降低，中心区保持可见但不可触发。

视觉意图：

- 默认态：中心按钮强调“可触发”，扇区保持低对比待机状态。
- 展开态：扇区边界清晰，选中扇区应有明显高亮与标签反差。
- 拖拽态：高亮跟随命中扇区移动，未命中区域不触发假高亮。
- 取消态：拖到外圈之外时高亮消失，松手不提交动作。
- 禁用态：中心按钮与扇区统一降饱和度，不出现交互反馈。

## 6. 状态覆盖矩阵

| 状态 / 能力 | 覆盖方式 | 预期结果 |
|---|---|---|
| 默认折叠态 | 初始化渲染 | 仅中心触发区突出，外圈动作项不抢主视觉 |
| 展开态 | 点击中心触发区 | 扇区菜单展开，所有动作项布局完整可读 |
| 悬停高亮态 | 从中心向某扇区拖拽 | 命中扇区高亮，非命中扇区恢复默认 |
| 松手确认态 | 在有效扇区内释放 | 选中动作生效，菜单收起，提示文案更新 |
| 取消态 | 拖到取消区或外圈后释放 | 不触发动作，菜单收起，保持上一次结果 |
| 禁用态 | egui_view_set_enable(..., false) | 点击和拖拽均无效，视觉反馈保持禁用样式 |
| 边界命中 | 拖拽经过扇区边界 | 高亮切换稳定，无闪烁、无越界命中 |
| 紧凑布局 | 小尺寸 4 扇区布局 | 图标/文本不重叠，扇区边界仍可识别 |
## 7. 录制动作设计（映射 egui_port_get_recording_action()）

建议用 EGUI_SIM_WAIT、EGUI_SIM_CLICK、EGUI_SIM_DRAG 组合录制：

| 序号 | 动作类型 | 目标 | 间隔 | 预期效果 |
|---:|---|---|---:|---|
| 0 | WAIT | 初始界面稳定 | 400ms | 记录默认折叠态 |
| 1 | CLICK | radial_primary 中心触发区 | 700ms | 主菜单展开 |
| 2 | DRAG | 从主菜单中心拖到右上扇区 | 500ms | 高亮目标动作并在释放后确认选择 |
| 3 | CLICK | radial_primary 中心触发区 | 700ms | 主菜单再次展开 |
| 4 | DRAG | 从主菜单中心拖到外圈取消区 | 400ms | 菜单收起且不改变已选结果 |
| 5 | CLICK | radial_compact 中心触发区 | 700ms | 紧凑菜单展开 |
| 6 | DRAG | 从紧凑菜单中心拖到左侧扇区 | 400ms | 验证小尺寸扇区命中与确认 |
| 7 | CLICK | radial_disabled 中心触发区 | 700ms | 禁用态保持无变化 |
| 8 | WAIT | 尾帧稳定 | 800ms | 收集稳定回归帧 |

实现时建议：

- 优先通过 view 中心点和相对坐标计算拖拽起止点，避免写死屏幕坐标。
- 若展开动画存在延迟，可在首次展开后追加一次 WAIT 或 recording_request_snapshot()。
- 拖拽步数建议 10~14，保证扇区切换过程具有足够可见的交互帧。

## 8. 编译、runtime 与截图验收标准

建议命令：

make all APP=HelloCustomWidgets APP_SUB=navigation/radial_menu PORT=pc
python scripts/code_runtime_check.py --app HelloCustomWidgets --app-sub navigation/radial_menu --timeout 10 --keep-screenshots

关键截图输出目录：

- runtime_check_output/HelloCustomWidgets_navigation/radial_menu/default/

通过条件：

1. example/HelloCustomWidgets/navigation/radial_menu/ 下的头文件、源文件、示例代码都存在且可构建。
2. 运行时检查通过，应用日志中没有 [RUNTIME_CHECK_FAIL]。
3. 截图非空白，扇区展开与选择反馈可见。
4. 交互验证通过，至少能观察到 2 次有效状态变化。
5. runtime 完成后，要把关键截图直接贴到对话框，不再只在本地目录查看。
6. 递归质量迭代次数不少于 30 次。
7. 控件收口时，需先更新追踪表，再单独提交一次该控件的 commit。
## 9. 已知限制与下一轮迭代计划

已知限制（启动阶段预期）：

- 第一版可先支持纯色扇区 + 文字，不强制首版就支持复杂图标。
- 第一版可先做单层菜单，不立即引入二级子菜单。
- 动画优先级低于命中准确性与回归稳定性。

下一轮迭代计划：

1. 增加图标 + 文字混排能力，并支持不同扇区的最小文本宽度策略。
2. 支持中心按钮自定义图标、标题和副标题。
3. 为旋钮 / 键盘输入增加角度切换与确认逻辑。
4. 增加展开 / 收起过渡动画，并验证 runtime 截图稳定性。

## 10. 与现有控件的重叠分析与差异化计划

对比对象：

- menu
- button_matrix
- tab_bar
- compass

差异化边界：

- radial_menu 不是线性菜单替代品，而是方向驱动的上下文动作控件。
- radial_menu 不是页面导航控件，不承担 tab_bar 的页面切换语义。
- radial_menu 不是通用角度显示器，不承担 compass 的信息展示职责。
- radial_menu 的核心价值必须来自“中心触发 + 扇区命中 + 拖拽释放确认”的完整交互链路。

本轮明确的非重叠目标：

1. 优先验证极坐标命中测试，而不是复用矩形按钮命中逻辑。
2. 选中反馈必须围绕扇区展开，不退化成列表高亮。
3. 示例场景聚焦“临时上下文动作面板”，不做页面切换型 demo。
