# RadarChart 控件设计说明

## 1. 为什么需要这个控件

radar_chart 用于表达“一个对象在多个维度上的综合能力轮廓”，适合展示对比型、均衡性导向的数据，而不仅仅是单轴趋势或单值进度。

它适合这些场景：

- 设备状态总览，例如 CPU / RAM / IO / TEMP / NET 五个维度的负载轮廓。
- 角色、车型、方案、传感器模组等多维评分对比。
- 需要快速判断“偏科”还是“均衡”，而不是只看单项最高值。
- 在 240x320 这类小屏幕里，以更紧凑、更具辨识度的方式呈现多维指标。

本控件的价值不是“再做一种图表皮肤”，而是引入一种新的多维极坐标数据表达方式，为后续 1000 控件计划补齐具有视觉辨识度的数据控件样板。

## 2. 为什么现有控件还不够

现有控件不能直接替代 radar_chart：

- chart_line：更适合时间序列或顺序序列，不适合表达多维均衡关系。
- chart_bar：擅长单维度并排比较，但不适合看整体轮廓。
- chart_pie：适合份额占比，不适合展示独立维度能力上限。
- chart_scatter：更适合离散点分布，不适合作为封闭能力轮廓图。
- compass：虽是极坐标视觉，但核心是方向展示，不是多维数据面图。

因此 radar_chart 的差异化边界是：

1. 多个固定维度均匀分布在极坐标轴上。
2. 数据通过闭合折线/面积表现整体轮廓。
3. 支持至少两组数据叠加对比，并保留层次清晰的网格参考。

## 3. 应用概览与使用场景

本次 HelloCustomWidgets 示例计划展示 3 个区域：

- radar_primary：主示例，展示 6 维数据的面积图与折线边界。
- radar_compare：对比示例，展示两组数据叠加的差异。
- radar_mini：紧凑示例，用于验证小尺寸下的可读性。

目标场景：

- 系统诊断：多个性能维度的均衡性。
- 产品对比：方案 A / 方案 B 多维评分对比。
- 设备画像：传感器、模块、算法能力雷达图。

控件目录：

- example/HelloCustomWidgets/chart/radar_chart/

## 4. 视觉与布局规格

屏幕基准：240 x 320。

整体布局建议：

- 顶部标题：220 x 18。
- 主图区域 radar_primary：168 x 168，6 轴，4 层背景网格。
- 对比图区域 radar_compare：168 x 120，6 轴，2 组对比轮廓。
- 小图区域 radar_mini：92 x 92，5 轴，1 组简化轮廓。

视觉意图：

- 背景网格要弱，不抢数据层主体。
- 当前主数据层要有填充面 + 边界线双重表达。
- 对比数据层要有低透明度填充或只画边界，避免覆盖主层。
- 维度标签优先短词，必要时使用缩写。
- 小尺寸版本允许减少网格层数，但不能丢失维度方向感。

## 6. 状态覆盖矩阵

| 状态 / 能力 | 覆盖方式 | 预期结果 |
|---|---|---|
| 默认态 | 初始渲染 | 网格、轴线、标签、数据面完整可见 |
| 单组数据态 | 只展示主数据 | 填充面与边界线都清晰 |
| 双组对比态 | 增加第二组数据 | 两组轮廓可区分，不糊成一片 |
| 小尺寸态 | mini 图区域 | 轴线、轮廓、标签仍具辨识度 |
| 边界值态 | 0 和最大值混合 | 轮廓不越界、不塌陷、不闪烁 |
| 动态切换态 | 录制中切换数据模式 | 状态提示更新，图形变化可见 |

| 序号 | 动作类型 | 目标 | 间隔 | 预期效果 |

| 0 | WAIT | 初始界面稳定 | 400ms | 记录默认态 |

## 8. 编译、runtime 与截图验收标准

建议命令：

make all APP=HelloCustomWidgets APP_SUB=chart/radar_chart PORT=pc
python scripts/code_runtime_check.py --app HelloCustomWidgets --app-sub chart/radar_chart --timeout 10 --keep-screenshots

关键截图输出目录：

- runtime_check_output/HelloCustomWidgets_chart/radar_chart/default/

通过条件：

1. radar_chart 头文件、源文件、示例代码都可构建。
2. runtime 通过，日志中无 [RUNTIME_CHECK_FAIL]。
3. 网格、轴线、主数据层、对比层在截图中都可辨认。
4. 小尺寸版本不出现严重重叠或不可读。
5. runtime 后关键截图直接贴到对话框中。
6. 迭代次数不少于 30 次。

## 9. 已知限制与下一轮迭代计划

已知限制（启动阶段预期）：

- 第一版可先支持固定维度数量，不立即做运行时动态增减轴。
- 第一版允许标签使用英文短词，暂不做复杂文本排版。
- 第一版先做点击切换数据，不急着上动画过渡。

下一轮迭代计划：

1. 增加平滑过渡动画，让轮廓切换更柔和。
2. 支持每个维度的阈值圈或高亮区段。
3. 支持更多数据组的显示策略，例如 hover 高亮某一组。
4. 支持更细的主题配色与背景网格样式。

## 10. 与现有控件的重叠分析与差异化计划

对比对象：

- chart_line
- chart_bar
- chart_pie
- chart_scatter
- compass

差异化边界：

- radar_chart 不是线性图表，不承担时间序列趋势展示职责。
- radar_chart 不是占比图，不承担份额分配表达职责。
- radar_chart 不是方向仪表，不承担航向或方位展示职责。
- radar_chart 的核心价值在于“多维极坐标 + 闭合轮廓 + 多组对比”。

本轮明确的非重叠目标：

1. 主数据层必须形成完整闭合面，不退化为普通折线图。
2. 背景网格必须服务于多维读数，而不是仅作装饰。
3. 对比示例必须能看出两组轮廓之间的形态差异。
