# PatternLock 示例

## 0. 为什么设计这个控件（设计动机）
`pattern_lock` 面向“轻量安全确认 + 手势路径触发”场景。
在嵌入式设备上，完整键盘输入通常成本高、步骤多，而图案手势可以在小屏里快速完成身份确认或功能触发。
该控件把“连续路径输入”作为一等语义，而不是离散按钮组合。

## 0.1 为什么不直接用现有控件
- `button_matrix`：只适合离散点击，不支持连续拖拽路径语义。
- `slider/arc_slider`：是连续数值输入，不具备节点序列表达能力。
- `switch/toggle_button`：只能表示二值状态，无法承载路径校验。

结论：现有控件无法直接替代“路径序列 + 最短长度校验 + 错误反馈”的组合能力。

## 1. 示例概览与使用场景
本示例展示 3x3 手势图案锁：
- 拖拽节点生成图案。
- 抬起后触发完成回调与完成通知回调。
- 节点数不足时显示错误颜色与错误文本。
- 通过 `Clear Pattern` 按钮清空状态。

典型场景：
- 本地设置页快速确认。
- 工程模式/维护模式入口。
- 游戏隐藏指令与快捷手势。

## 2. 视觉与布局规范
- 根容器：`228x300`，垂直居中布局。
- PatternLock 区域：`216x216`。
- 状态文本：`216x28`。
- 清空按钮：`140x34`。
- 节点布局：3x3 等分网格。
- 颜色策略：
  - 普通节点：`track_off`
  - 激活节点：`primary_light`
  - 路径：`secondary`
  - 错误：`danger`

## 3. 控件清单
| 变量名 | 类型 | 尺寸 | 初始状态 | 作用 |
|---|---|---|---|---|
| `pattern_lock_view` | `egui_view_pattern_lock_t` | 216x216 | `min_nodes=3, touch_expand=7` | 手势路径输入 |
| `status_label` | `egui_view_label_t` | 216x28 | 默认提示文本 | 展示成功/错误信息 |
| `clear_button` | `egui_view_button_t` | 140x34 | 可点击 | 清空当前图案 |

## 4. 状态覆盖矩阵
| 状态 | 覆盖说明 |
|---|---|
| idle | 初始提示文本可见 |
| tracking | 拖拽中显示路径与尾线 |
| completed(valid) | 输出节点序列并显示成功文本 |
| completed(invalid) | 显示错误颜色与错误文本 |
| disabled | 支持外部禁用，自动降饱和 |

## 5. 录制动作表（`egui_port_get_recording_action()`）
1. 拖拽 `0 -> 2`：验证有效路径。
2. 点击节点 `0`：验证无效路径（节点数不足）。
3. 点击 `Clear Pattern`：验证清空行为。
4. 拖拽 `0 -> 8`：验证对角桥接（自动补中点）。
5. 等待 500ms：稳定截图。

## 6. Runtime + Baseline 验收标准
- Runtime check 通过，且日志无 `[RUNTIME_CHECK_FAIL]`。
- Interaction 校验通过（变化帧数量达到阈值）。
- `visual_report` 非空、非空白帧。
- 首次仅首验；显式 `--promote-baseline` 后进入回归模式。
- 回归模式下 SSIM 达到阈值（默认 `0.92`）。

## 7. 已知限制与后续计划
- 当前固定 3x3 网格，后续可扩展 4x4。
- 回调默认返回节点数量，业务侧可通过 getter 获取序列。
- 可继续扩展超时自动清空和防暴力策略。

## 8. 与现有控件重叠分析与差异化边界
- 重叠点：都依赖基础触摸输入与基础绘制能力。
- 差异边界：
  - 连续路径语义（非离散点击）。
  - 自动桥接中间节点。
  - 路径长度校验与错误态反馈。
  - 可读的节点序列输出。

## 9. 迭代计划（>=10轮）
每轮必须记录：
- 视觉验证结论。
- 交互验证结论。
- 变更文件证据与运行报告证据。

## 10. 功能重复度评估
- 重复度：`medium`。
- 是否必要：是。
- 保留理由：图案路径交互是现有控件无法组合出的独立输入模型。
