# Autocomplete 示例

## 0. 为什么设计这个控件

`autocomplete` 用于“输入后建议选择”场景，控件应暴露“建议项”语义，而不是泛化的下拉选择语义。  
虽然底层可复用 `combobox`，但单独定义控件可以降低上层对 `combobox` 细节的耦合，并在 UI schema 与业务代码里保持意图清晰。

## 0.1 为什么不用现有控件

- `combobox`：交互接近，但 API 命名偏通用选择，不突出建议补全语义。  
- `textinput`：只负责输入，不提供建议列表展开与选择闭环。  
- `list/menu`：可展示选项，但无法独立表达“折叠输入框 + 展开建议 + 回填选择”的单控件生命周期。

设计取舍：复用 `combobox` 的渲染和基础交互，新增 `autocomplete` 语义边界和接口。

## 1. 概述

本示例验证基于 `combobox` 交互模型封装的轻量 `autocomplete` 控件。

## 2. 视觉与布局规范

- 主控件尺寸：`210x34`  
- 位置：根视图中居中  
- 建议列表：展开下拉，限制最大可见项

## 3. 控件清单

| 变量名 | 类型 | 尺寸 | 初始状态 | 目的 |
|---|---|---|---|---|
| `autocomplete_view` | `egui_view_autocomplete_t` | 210x34 | `index=0` | 建议项选择 |

## 4. 状态覆盖矩阵

| 状态 | 覆盖说明 |
|---|---|
| normal | 默认折叠态 |
| expanded | 点击后展开建议列表 |
| selected | 选择建议项后更新索引 |
| boundary | 重复展开/选择路径 |

## 5. 录制动作

`egui_port_get_recording_action()` 覆盖：

1. 展开下拉。  
2. 选择第 3 个建议项。  
3. 再次展开下拉。  
4. 选择第 1 个建议项。

## 6. 验收标准

- Runtime check 通过。  
- 帧序列中交互变化可见。  
- baseline 晋升后，回归检查通过。

## 7. 已知限制与下一步

- 当前使用静态建议项列表。  
- 动态过滤（基于输入文本）为下一步优化项。  
- 状态文本保持简洁，便于稳定录制与回归对比。

## 8. 功能重叠与差异化边界

- 与 `combobox` 在交互核心上存在重叠。  
- 差异化目标是“自动建议补全”场景语义与 API 命名，避免业务层误用成普通下拉框。

## 9. 回归录制配置说明

- 推荐稳定参数：`clock-scale=6`、`snapshot-settle-ms=40`、`snapshot-stable-cycles=2`。  
- 交互验收要求展开/选择流程至少产生 2 次有效帧变化。
